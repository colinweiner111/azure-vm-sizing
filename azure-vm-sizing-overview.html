How Azure Organizes Virtual Machines

Type / Category
(General purpose / Memory / Compute / GPU / HPC)
      ↓
Family
(D / E / F / N / H)
      ↓
Series
(Dsv5 / Dasv6 / Ebdsv5)
      ↓
Size (SKU)
(D16ds_v5 / D32as_v6)
  • Type / category = the workload class you're optimizing for (balanced, memory, compute, GPU, HPC)
  • Family = workload archetype (balanced, memory-optimized, GPU…)
  • Series = a lineup built on the same generation + capability pattern
  • Size = a specific deployable SKU with exact vCPU/memory and flags

How Microsoft Organizes VM Sizes in Documentation

Microsoft documentation reflects the same hierarchy you just saw — but it's split across different pages and navigation paths. 

Microsoft Docs / Portal
        │
        ├─ Family pages
        │    (workload intent)
        │
        ├─ Series pages
        │    (hardware + capabilities)
        │
        └─ Size tables
             (deployable SKUs + limits)
  • Family pages describe the workload focus (general purpose, memory, compute, GPU)
  • Series pages define hardware generations, capabilities, and supported features
  • Size tables list the deployable SKUs, limits, and regional availability

This is why you often see separate links for Series and Sizes in Microsoft Learn and the Azure portal — they answer different questions.

So… why is there a "Series" and "Size" link?

  • Series view answers: "What family lineup should I be in?" Hardware generation + capability pattern + intended workloads.
  • Size view answers: "Which exact SKU do I deploy?" vCPU count, memory amount, flags, accelerator model, version.
Important: Series and Size are in the same family but answer different questions — hardware generation vs deployment SKU. Always check both when designing solutions or troubleshooting sizing issues.

Series = "What hardware + capabilities?"

  • The Series describes the capability pattern + version.
  • It tells you about hardware features like:
    • CPU generation (Intel vs AMD vs Ampere)
    • Support for accelerators (GPU / FPGA / special compute)
    • Disk support (ephemeral / Premium / Ultra / local NVMe)
    • Networking capabilities (accelerated networking / RDMA / InfiniBand)
    • Version history (v5 vs v6)
Example: Dsv5 means D family (balanced/general-purpose), with Premium SSD support (s), version 5 hardware. Every size in the Dsv5 series shares this capability baseline.

Size = "How much hardware?"

  • The Size is the deployable SKU — the exact VM you provision.
  • It specifies:
    • vCPU count
    • Memory amount
    • Temporary disk size (if applicable)
    • Data disk limits
    • NIC limits
    • Accelerator model (for GPU VMs)
Example: D16s_v5 = 16 vCPUs, 64 GiB RAM, no local temp disk (no d flag), Dsv5 capabilities.

Concrete example: Dsv5 (series) vs D16s_v5 (size)

Series (Family + capabilities) Size (Deployable SKU)

Series tells you what kind of machine. Size tells you how big the machine is.

Why "series" and "size" both matter

  • Series determines compatibility and capabilities:
    • Can this run in my region?
    • Does it support the disk / network features I need?
    • What CPU generation am I getting?
  • Size determines scale and quotas:
    • Do I have enough quota for this many vCPUs?
    • Will this fit my memory / IOPS requirements?
    • Can I attach enough data disks / NICs?
Common mistake: Choosing a size without checking the series first. You might pick the right vCPU count but miss that the series doesn't support Premium SSD, or isn't available in your region.

How to pick a VM: start with series, then size

  1. Identify workload pattern (general, memory, compute, GPU, HPC)
  2. Choose a family (D, E, F, N, H…)
  3. Pick a series within that family (Dsv5, Easv6…)
    • Check hardware generation, CPU type, feature support
  4. Select a size (D16s_v5, E32as_v6…)
    • Match vCPU, memory, disk, and network needs
  5. Validate
    • Regional availability
    • Quota limits
    • Cost

Workload example 1: web application tier

Pattern

  • Moderate CPU and memory
  • Needs to scale horizontally
  • Benefits from Premium SSD for consistent IOPS

Sizing guidance

  • Start with D-series (general-purpose).
  • Pick s variant for Premium SSD support (Dsv5).
  • Size based on expected load: D4s_v5 → D8s_v5 → D16s_v5.
  • Use VMSS for horizontal scaling and availability.

Workload example 2: SQL Server database

Pattern

  • Memory-intensive (buffer pool + query cache)
  • High IOPS and throughput for transaction logs and data files
  • May need local temp disk for tempdb

Sizing guidance

  • Use E-series (memory-optimized) for SQL workloads.
  • Choose ds variant: local temp disk (d) + Premium SSD (s).
  • Example: E16ds_v5 (16 vCPUs, 128 GiB RAM, local temp disk).
  • Validate IOPS/throughput limits against your transaction log + data file requirements.

How hardware versions work (v4 → v5 → v6)

  • Version number indicates hardware generation:
    • v4: Intel Cascade Lake / AMD EPYC 2nd Gen
    • v5: Intel Ice Lake / AMD EPYC 3rd Gen
    • v6: Intel Sapphire Rapids / AMD EPYC 4th Gen / Ampere Altra
  • Higher version = newer CPU, better performance/efficiency, sometimes lower cost.
  • Not all regions have all versions yet.
Current state (late 2024/2025): v6 generation is still rolling out across regions. Check regional availability before planning v6 migrations. v5 remains widely available and stable.

Series naming: what's in the series name vs size name?

Series:  Dsv5
Size:    D16s_v5
  • Series groups the lineup: family + capability pattern + version (Dsv5).
  • Size adds the concrete deployable SKU: vCPU count + any extra flags (D16s_v5).

Docs show both because engineers often start at the series level ("Which lineup?") before picking a size ("How big?").

Workload example 3: AVD / VDI / graphics workloads

Pattern

  • GPU acceleration needed for UX, graphics apps, or media workloads
  • Driver + image compatibility matters as much as vCPU/memory

Sizing guidance

  • Choose NV series for visualization/graphics-optimized workloads.
  • Validate GPU model availability (accelerator type is often in the SKU for newer GPU sizes).
  • Plan driver lifecycle: version pinning, monitoring, and rollback.

Workload example 4: HPC / MPI / low-latency clusters

Pattern

  • Scale-out compute, tightly coupled nodes, sensitive to latency/jitter

Sizing guidance

  • Look for r (RDMA / InfiniBand secondary network) or HPC families (H*).
  • Placement and network fabric are part of the performance model.
  • Don't treat these as "just bigger VMs." They are a different platform tier.
